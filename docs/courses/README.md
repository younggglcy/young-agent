# Courses

`docs/courses/` 用来沉淀教学、学习向的内容。它的读者首先是人：
希望几年后再回来看，仍然能沿着一条清楚的路径理解我们怎样一步一步
从 0 到 1 构建出属于自己的 agent。

## 定位

课程不是临时笔记，也不是实现清单。课程要把真实构建过程整理成可连续
阅读的学习路径：

- 从具体问题开始，而不是从抽象名词开始。
- 在第一次使用概念前先解释它，例如 Agent Kernel、Surface、Capability Pack。
- 把架构判断、代码产物、测试验证和踩坑经验放到同一个叙事里。
- 链接真实仓库文件，让读者能从课程跳到当前项目事实。
- 保留“当时为什么这么做”，而不只是记录“最后做成了什么”。

## 与其他文档的边界

- `docs/prd/`：说明要做什么、为什么做、完成标准是什么。
- `docs/adr/`：记录已经接受的架构决策和后果。
- `docs/issues/`：把工作拆成 agent 可以执行的小任务。
- `docs/lessons/`：保存可独立阅读的经验、原则和复盘。
- `docs/courses/`：把这些材料串成面向学习者的顺序课程。

如果一个经验很短，先放进 `docs/lessons/`。如果它开始变成一条需要
循序渐进阅读的路径，就把它整理进 `docs/courses/`。

## 维护约定

- 每完成一个重要里程碑，更新对应课程的目录或课程日志。
- 课程章节优先解释“问题 -> 判断 -> 产物 -> 验证 -> 下一步”。
- 不假设读者已经熟悉本仓库；必要术语要在本章内重新引入。
- 不为了完整而堆材料；宁可短而清楚，也不要把 PRD、ADR 原文搬进来。
- 当实现和课程内容出现偏差时，以当前仓库事实为准并修订课程。
